PROJECT SUMMARY (peled-index)
=============================

Purpose:
Peled Index is an author-level academic scoring system. It takes a list of researchers (with their publication titles and PhD start years), enriches each paper via the Crossref API and SCImago journal data, computes a per-author score normalized by career length, and serves the results through a web UI where users can search authors and view percentile-rank plots within scientific fields. Users can also search and scrape Google Scholar profiles on demand to score authors not in the precomputed dataset.

Deployment:
The app is deployed on Railway (peled-index-production.up.railway.app). The Procfile runs uvicorn on web.app:app. Railway environment variables: SERPAPI_KEY, ADMIN_KEY, DATABASE_URL (from Railway Postgres addon).

Architecture:
The project has four layers:
  1. core/ — scoring pipeline and helpers (offline batch processing + on-demand Scholar scoring)
  2. web/ — FastAPI serving layer with API routes, handlers, database persistence, and static frontend
  3. data/ — precomputed results JSON files + SCImago CSV (read-only at runtime)
  4. local/ — legacy duplicates of core files + an upload utility script

Data loading:
  - On startup (lifespan in web/app.py), the app calls init_db() to connect to Postgres (graceful no-op if DATABASE_URL is not set), then calls load_results_into_state(app) which reads the precomputed JSON from disk AND scholar results from Postgres, merges them (scholar entries override on collision), and stores the merged list in app.state.results and app.state.index.
  - Scholar scrapes are persisted to Postgres via upsert_scholar_result() in web/db.py. This survives restarts, redeployments, and is shared across all dynos/instances.
  - An admin reload endpoint (POST /api/admin/reload, protected by ADMIN_KEY header) re-reads both data sources into memory without restarting the app.
  - An admin scholar-results endpoint (GET /api/admin/scholar-results) returns all scholar-scraped entries from the database.
  - The precomputed JSON file (data/phd_isr_res_filtered.json) is deployed as part of the git repo. To update it: replace the file, push, and optionally hit the reload endpoint.

File-by-file breakdown:

  core/helpers.py (~14 KB)
    The low-level workhorse module. Responsibilities:
    - Crossref API interaction: fetches candidate records for a title+author query, with retry/backoff on 429s. Uses a persistent requests.Session.
    - Text normalization: heavy normalization pipeline (ftfy, html unescape, unicode NFKD, ASCII folding, lowercasing, punctuation removal) used for title and author name matching. Title matching uses SequenceMatcher with a 0.9 threshold.
    - ISSN normalization and SCImago CSV parsing: loads scimagojr2024.csv (semicolon-delimited) into two dicts keyed by normalized ISSN — one for SJR values, one for mapped high-level fields.
    - SCImago area-to-field mapping: a hardcoded dict (SCIMAGO_AREA_TO_FIELD) that collapses ~25 SCImago subject areas into 4 broad fields: Life Sciences, Physics and Chemistry, Computer Science and Engineering, Social Sciences and Humanities.
    - Crossref record extraction: functions to extract title, ISSNs, year, venue, citations, first author name, preprint detection (by type/subtype + platform name matching against biorxiv/medrxiv/chemrxiv/arxiv).
    - Match selection logic (find_crossref_match): for a given title, tries top-1 hit first, then next 3. Requires title similarity >= 0.9 + author name match + DOI present. Prefers journal-article type, falls back to first non-article match.
    - Paper deduplication by title.
    - JSON I/O for author list input and results output.
    - Field ranking: rank_fields_by_paper_count returns fields sorted by paper count descending.

  core/pipeline.py (~5.7 KB)
    Orchestration and scoring logic. Responsibilities:
    - build_papers_from_titles: for one author, fetches all papers concurrently (ThreadPoolExecutor, 8 workers) via find_crossref_match, enriches each with SJR and field data.
    - Parsing helpers: parse_year, parse_int, parse_float — safe converters returning None or 0 on failure.
    - compute_paper_score: per-paper score formula:
        * Excludes papers with no year, preprints, or future years.
        * citations_per_year = citations / paper_age (where paper_age = current_year - paper_year + 1).
        * authorship_weight = 1.0 if first author, else 0.1 (non_first_author_weight).
        * If journal SJR available: score = authorship_weight * (0.3 * citations_per_year + 0.7 * journal_sjr).
        * If no SJR: score = authorship_weight * citations_per_year.
    - infer_start_year_from_papers: infers career start from the earliest publication year (used for Scholar-scraped authors who have no input start_year).
    - compute_author_score: sums paper scores, divides by active_years (current_year - start_year + 1). Returns 0.0 if no valid papers or non-positive active years or start_year is None.
    - compute_compressed_author_score: applies log1p compression to reduce tail influence.
    - score_author: ties together paper building, scoring, and field aggregation for one author from the input JSON (uses input start_year).
    - score_author_from_scholar: same flow but for Scholar-scraped authors — infers start_year from earliest paper, since Scholar profiles do not provide PhD start year.
    - run_pipeline: top-level entry point for batch processing. Loads SCImago data, iterates authors sequentially, scores each, saves results JSON.

  core/plots.py (~6.4 KB)
    Visualization module. Generates per-field score distribution histograms as Base64-encoded PNG images (matplotlib). Key functions:
    - filter_eligible_authors_by_field: filters to authors with score > 0 and matching field.
    - extract_author_scores: extracts score values from a results list.
    - apply_log1p_transform: applies log1p to reduce outlier noise in plots.
    - compute_percentile: fraction of log1p-transformed scores below the candidate.
    - plot_score_distribution_for_field: dark-themed histogram, dual-tone bins (darker left of candidate), dashed red candidate line with percentile annotation. Returns Base64 string.
    - generate_plots_for_author: orchestrates plot generation for all fields of a given author (used by CLI, not by web layer).

  core/scholar.py (~3.1 KB)
    Google Scholar integration via SerpAPI. Responsibilities:
    - fetch_scholar_page: fetches one page of articles from a Scholar author profile (100 per page).
    - extract_scholar_author_name, extract_scholar_institution, extract_scholar_paper_titles: parse SerpAPI response fields.
    - has_next_page: checks SerpAPI pagination for more pages.
    - deduplicate_titles: deduplicates by normalized title.
    - fetch_all_scholar_papers: paginates through all pages, returns (author_name, institution, deduplicated_titles).
    - search_scholar_by_name: searches Google Scholar for author name candidates, returns list of {author_id, name, affiliations, email, cited_by}.

  core/main.py (~2.9 KB)
    CLI entrypoint for batch pipeline runs. Currently hardcoded to local Windows paths. Calls run_pipeline. Contains commented-out utility functions.

  web/app.py (~3.7 KB)
    FastAPI application setup and admin endpoints. Responsibilities:
    - Data loading functions: _build_index (creates MD5-keyed lookup dict), _load_json_file (reads JSON from disk), _merge_results (merges precomputed + scholar, scholar overrides on collision), load_results_into_state (loads both sources into app.state).
    - Lifespan: calls init_db(), load_results_into_state(), loads SCImago data, sets current_year and serpapi_key on app.state.
    - GET /api/startup-info: returns filenames, precomputed count, scholar DB count, total loaded count.
    - POST /api/admin/reload: re-reads all data into app.state (protected by ADMIN_KEY header).
    - GET /api/admin/scholar-results: returns all scholar entries from Postgres (protected by ADMIN_KEY header).
    - Mounts static files for the frontend.

  web/db.py (~4.4 KB)
    Postgres persistence layer for scholar results. Gracefully degrades if DATABASE_URL is not set (all functions become no-ops, app still works with precomputed data only). Responsibilities:
    - Connection management: parses DATABASE_URL, handles postgres:// vs postgresql:// prefix, context manager for connections with commit/rollback.
    - init_db: creates scholar_results table if not exists, sets _db_available flag. Prints status to stdout.
    - upsert_scholar_result: inserts or updates one author entry (JSONB data column). No-op if DB unavailable.
    - load_all_scholar_results: returns all rows as dicts with _db_meta (author_id, created_at, updated_at). Empty list if DB unavailable.

    - Table schema: scholar_results (author_id TEXT PK, name TEXT, institution TEXT, data JSONB, created_at TIMESTAMP, updated_at TIMESTAMP).

  web/api/routes.py (~2.5 KB)

Data flow:
  1. BATCH (offline): Input JSON (array of {name, start_year, institution, publications}) -> Crossref enrichment (concurrent, 8 threads per author) -> SCImago SJR/field lookup -> paper scoring -> author scoring -> results JSON saved to data/.
  2. ON-DEMAND (live): User searches Scholar -> selects profile -> SerpAPI fetches all paper titles -> same Crossref+SCImago enrichment and scoring as batch -> result added to in-memory state AND persisted to Postgres.
  3. STARTUP: Precomputed JSON loaded from disk + scholar results loaded from Postgres -> merged (scholar overrides) -> stored in app.state.results and app.state.index.
  4. ADMIN RELOAD: POST /api/admin/reload re-executes step 3 without restart.

Scoring formula summary:
  paper_score = authorship_weight * (0.3 * (citations / paper_age) + 0.7 * journal_sjr)
    - Falls back to authorship_weight * (citations / paper_age) when SJR is unavailable.
    - authorship_weight: 1.0 for first author, 0.1 otherwise.
    - paper_age = current_year - paper_year + 1.
    - Preprints and papers with missing/future years are excluded.
  author_score = sum(paper_scores) / active_years
    - active_years = current_year - start_year + 1.
    - For Scholar-scraped authors: start_year is inferred from earliest publication year.
  author_score_log1p = log1p(author_score) — compressed version for reduced tail influence.

Key design decisions:
  - Functional programming paradigm. No classes. Stateless functions. Minimal mutability.
  - Microservice-style module separation: helpers is the extensive low-level interface, pipeline is orchestration, plots is visualization, scholar is Google Scholar integration, web is serving, db is persistence.
  - Crossref matching is strict: requires title similarity >= 0.9 AND author name match AND DOI present.
  - SCImago areas are collapsed into 4 broad fields for cross-discipline comparison.
  - The web layer serves precomputed results merged with on-demand Scholar results. Pipeline runs are offline/batch.
  - Scholar results are persisted to Postgres so they survive restarts and are shared across all instances.
  - The database layer degrades gracefully: if DATABASE_URL is not set, the app works with precomputed data only, and Scholar scrapes are in-memory only.
  - Root-level files do not exist. Legacy duplicates live under local/. The web layer imports from core/.
